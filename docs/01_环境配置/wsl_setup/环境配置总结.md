# MuJoCo 项目环境配置总结

## 运行环境

- **系统**: WSL2 (Ubuntu) on Windows 11
- **Python**: 3.11 (miniforge3/envs/dev)
- **显示系统**: WSLg (内置 GPU 加速支持)

## 遇到的问题及解决方案

### 问题 1: 依赖冲突

**错误信息**:
```
tdw 1.13.0.0 requires ikpy==3.1, but you have ikpy 3.4.2 which is incompatible.
```

**原因**: 
- 环境中已安装 `tdw` 包，要求 `ikpy==3.1`
- 本项目需要 `ikpy==3.4.2`

**解决方案**:
- 用户选择忽略 tdw（已不使用）
- 如需同时使用，应创建独立 conda 环境

### 问题 2: Qt 平台插件加载失败

**错误信息**:
```
qt.qpa.plugin: Could not load the Qt platform plugin "xcb" in "" even though it was found.
Available platform plugins are: eglfs, linuxfb, minimal, minimalegl, offscreen, vnc, wayland-egl, wayland, wayland-xcomposite-egl, wayland-xcomposite-glx, webgl, xcb.
已中止 (核心已转储)
```

**原因**: 
- Qt 的 xcb 平台插件缺少系统依赖库

**解决方案**:
安装 Qt xcb 所需的系统依赖：
```bash
sudo apt-get update && sudo apt-get install -y \

    libxcb-cursor0 \

    libxcb-icccm4 \
    libxcb-image0 \

    libdbus-1-3 \
    libxcb-xkb1 \
    libxkbcommon0 \
    libgl1-mesa-glx \
    libglib2.0-0
```

### 问题 3: 显示系统配置

**尝试过程**:

#### 方案 A: MobaXterm X11 转发（失败）
- 配置 `export DISPLAY=172.26.64.1:0.0`
- 设置 `export LIBGL_ALWAYS_INDIRECT=1`
- **结果**: `xeyes` 可以显示，但 MuJoCo 报错 `OpenGL version 1.5 or higher required`
- **原因**: MobaXterm X Server 的 OpenGL 支持有限，软件渲染版本过低

#### 方案 B: WSLg 原生支持（成功✅）
- 配置 `export DISPLAY=:0`
- **取消** `LIBGL_ALWAYS_INDIRECT` 环境变量
- **结果**: 程序正常运行，支持完整的 GPU 加速

**技术解析：为什么 `DISPLAY=:0` 可以正常运行？**

`DISPLAY` 环境变量指定了 X11 客户端（应用程序）应该连接到哪个 X Server。格式为 `[host]:display[.screen]`：

1. **`DISPLAY=:0` 的含义**：
   - `:0` 表示本地第一个显示服务器（省略主机名代表本地）
   - WSLg 在 WSL2 启动时自动创建了一个 Wayland/X11 混合显示服务器
   - 这个服务器监听在 Unix socket `/tmp/.X11-unix/X0` 上

2. **与 `DISPLAY=172.26.64.1:0.0` 的区别**：
   - `172.26.64.1:0.0` 是通过 TCP 网络连接到 Windows 宿主机的 X Server（MobaXterm）
   - 需要数据通过网络栈传输，延迟更高
   - MobaXterm 的 X Server 只提供软件 OpenGL 渲染（Mesa llvmpipe）
   - 最高支持 OpenGL 1.5，无法满足 MuJoCo 的 OpenGL 3.3+ 要求

3. **WSLg 的优势**：
   - **直接访问 GPU**：通过 `/dev/dxg` 设备，WSL2 可以直接访问 Windows 的 GPU（使用 DirectX 到 Vulkan/OpenGL 的转换层）
   - **硬件加速**：支持完整的现代 OpenGL（3.3+）和 Vulkan
   - **高性能**：使用共享内存和本地 IPC，而非 TCP 网络
   - **零配置**：Windows 11 的 WSL2 自动启动 WSLg，无需手动配置 X Server

4. **工作流程**：
   ```
   MuJoCo 应用 (OpenGL 调用)
        ↓
   DISPLAY=:0 → Unix Socket /tmp/.X11-unix/X0
        ↓
   WSLg (Xwayland + Weston Compositor)
        ↓
   /dev/dxg → Windows GPU Driver
        ↓
   物理 GPU 硬件加速渲染
        ↓
   Windows 桌面显示
   ```

5. **为什么必须取消 `LIBGL_ALWAYS_INDIRECT`**：
   - 这个变量强制使用间接渲染（Indirect GLX）
   - 间接渲染会将 OpenGL 命令发送到远程 X Server 执行
   - 即使连接到 WSLg，也会绕过本地 GPU 加速
   - 导致回退到软件渲染，无法满足性能和版本要求

```bash
unset LIBGL_ALWAYS_INDIRECT  # 重要：取消间接渲染
```

## 环境变量持久化

在 `~/.bashrc` 中已添加：
```bash
# X11 forwarding for MobaXterm (override WSLg)
if command -v ip &> /dev/null; then
    export DISPLAY=$(ip route show | grep -i default | awk '{print $3}'):0.0
    export LIBGL_ALWAYS_INDIRECT=1
fi
```

**注意**: 由于最终使用 WSLg，上述配置会导致冲突。建议注释掉或修改为：
```bash
# Use WSLg for GUI applications
export DISPLAY=:0
# Do NOT set LIBGL_ALWAYS_INDIRECT for GPU acceleration
```

## 关键经验总结

1. **WSLg vs X11 转发**:
   - **WSLg**：支持完整的 GPU 加速和现代 OpenGL，通过 `/dev/dxg` 直接访问 Windows GPU
   - **X11 转发（MobaXterm）**：适合简单 GUI，但只提供软件 OpenGL 渲染（最高 1.5）
   - **选择原则**：需要 3D 渲染或 GPU 计算的应用必须使用 WSLg

2. **DISPLAY 环境变量详解**:
   - `DISPLAY=:0`：连接到本地 WSLg 服务器（Unix socket），支持硬件加速
   - `DISPLAY=<IP>:0.0`：通过 TCP 连接到远程 X Server，只能软件渲染
   - **格式**: `[hostname]:display[.screen]`，省略 hostname 表示本地连接

3. **LIBGL_ALWAYS_INDIRECT 的影响**:
   - 强制使用间接渲染（Indirect GLX），将 OpenGL 命令发送到 X Server 执行
   - 即使连接 WSLg 也会绕过本地 GPU，回退到软件渲染
   - **结论**：对于 MuJoCo 这种需要 OpenGL 3.3+ 的应用，**必须取消**此变量

4. **依赖管理**:
   - Qt 应用需要完整的 xcb 系统库（`libxcb-*`）
   - Python 包只是绑定，底层依赖需单独安装
   - 使用 `dpkg -l | grep libxcb` 验证安装状态

5. **WSL2 GUI 要求**:
   - Windows 11 或 Windows 10 21H2+
   - WSL 版本 2.0.0+
   - 已自动配置 WSLg（检查：`ps aux | grep Xwayland`）

6. **性能优化**:
   - WSLg 使用共享内存和 Unix socket，比 TCP X11 转发快 10-100 倍
   - GPU 直通技术允许 WSL 应用直接使用 Windows GPU，无性能损失
   - 适合运行仿真、渲染、深度学习等 GPU 密集型应用

## 故障排查

### 如果出现 "Could not load Qt platform plugin"
```bash
# 检查是否安装了 xcb 库
dpkg -l | grep libxcb

# 重新安装
./install_qt_deps.sh
```

### 如果出现 "OpenGL version required"
```bash
# 确保未设置间接渲染
unset LIBGL_ALWAYS_INDIRECT

# 检查 OpenGL 版本
glxinfo | grep "OpenGL version"

# 确认使用 WSLg
echo $DISPLAY  # 应该显示 :0

# 验证 GPU 加速状态
glxinfo | grep "direct rendering"  # 应该显示 "direct rendering: Yes"
glxinfo | grep "OpenGL renderer"   # 应该显示你的显卡型号，而非 "llvmpipe" 或 "software"
```

**诊断说明**：
- `direct rendering: No` + `llvmpipe` → 正在使用软件渲染，检查 `LIBGL_ALWAYS_INDIRECT` 和 `DISPLAY`
- `OpenGL version` < 3.3 → X Server 不支持硬件加速，切换到 WSLg
- 如果 `glxinfo` 未安装：`sudo apt install mesa-utils`

### 如果窗口无法显示
```bash
# 检查 WSLg 状态
ps aux | grep Xwayland

# 测试 X11
xeyes

# 重启 WSL
# 在 Windows PowerShell 中执行:
# wsl --shutdown
```

## 附录：技术概念速查

### WSLg 工作原理
```
应用程序 (Linux)
    ↓ X11/Wayland 协议
Xwayland/Weston (WSL2 内部)
    ↓ 共享内存 + IPC
RDP 协议栈
    ↓ Hyper-V 虚拟通道
Windows 桌面窗口管理器
    ↓
物理显示器
```

### GPU 加速路径
```
OpenGL 调用 (应用)
    ↓
Mesa 3D 驱动 (WSL2)
    ↓
/dev/dxg 设备 (GPU 直通)
    ↓
Windows DirectX 运行时
    ↓
物理 GPU 硬件
```

### 常见 DISPLAY 值对比
| DISPLAY 值 | 连接方式 | 性能 | GPU 加速 | 使用场景 |
|-----------|---------|------|---------|---------|
| `:0` | Unix socket | 最快 | ✅ 是 | WSLg 本地应用 |
| `localhost:0` | TCP loopback | 中等 | ❌ 否 | 测试/调试 |
| `<IP>:0.0` | TCP 网络 | 慢 | ❌ 否 | 远程 X11 转发 |
| 未设置 | - | - | - | 无法启动 GUI |
